# 08-test-data-design: テストデータ設計

## 目的

E2Eテストに必要なテストデータの準備・初期化・後片付け方法を定義する。現行環境と移行後環境で同一の前提データを用意できるようにする。

## 担当Phase

Phase 8: テストデータ設計（インクリメンタルに更新）

## 実施タイミング

機能IDごとのループ内で、07-architecture-design の後、09 の前に毎回実施する。
ただし毎回フルで実施するのではなく、**差分更新**を行う。

- **初回**: 最初の機能IDの07-architecture-design完了後に骨格を作成する。テストユーザー定義・初期化方針・後片付け方針など、全機能で共通する基盤部分を設計する。この時点で確認済みの機能に加え、一般的なWebアプリケーションのテストデータ設計の知見（ロール別テストユーザー、境界値、マスターデータとトランザクションデータの分離、テスト間独立性の確保パターン等）を踏まえて設計する。
- **2回目以降**: 新しい機能のUI仕様・API仕様・ユースケースを確認し、既存のテストデータ設計で対応できるか判断する。新しいテストユーザー・マスターデータ・データバリエーションが必要な場合は追記・修正する。レビューの結果、変更不要と判断した場合は「レビュー済み・変更なし」として記録する。

## 実行手順

### Step 0: 事前確認・既存設計の確認

1. CLAUDE.md を読む。
2. spec/worklog/F-XX-YY.md を読み、前回の作業状況を確認する。
3. spec/traceability/F-XX-YY.md を読み、ステータスを確認する。
4. 対象機能IDとスコープを確認する。

2回目以降は spec/test-data.md を読み、現在のテストデータ設計を確認する。今回の対象機能に、既存の設計でカバーできない新しいテストユーザー・マスターデータ・データバリエーションが必要かを判断する。レビューの結果、変更不要と判断した場合は「レビュー済み・変更なし」としてworklogに記録し、Step 8（spec/test-data.md の作成）以降に進む。

### Step 1: テストユーザーの整理

各ロール/ユーザー種別に必要なテストユーザーを定義する:

```markdown
## テストユーザー

| ユーザーID | ユーザー名 | ロール | 用途 | 備考 |
|---|---|---|---|---|
| test-admin | テスト管理者 | admin | 管理機能テスト | |
| test-user | テスト一般ユーザー | user | 一般機能テスト | |
```

### Step 2: 初期データの定義

E2Eテスト実行前に必要なデータを定義する:

```markdown
## 初期データ

### マスターデータ
| テーブル/エンティティ | データ件数 | 内容 | 投入方法 | 機能ID |

### トランザクションデータ
| テーブル/エンティティ | データ件数 | 内容 | 投入方法 | 機能ID |
```

### Step 3: データ投入方法の選定

以下の方法から、各データに適した投入方法を選定する:

- **DB直接投入**: SQL / seed script
  - メリット: 高速、確実
  - デメリット: DBスキーマに依存、移行後にスキーマが変わると修正必要
- **API投入**: APIを使ったデータ作成
  - メリット: ビジネスロジックを通る、スキーマ変更に比較的強い
  - デメリット: APIが未実装だと使えない、低速
- **画面操作投入**: Playwright で画面操作
  - メリット: 最も実際の利用に近い
  - デメリット: 低速、画面変更に脆弱

選定理由を記録する。

### Step 4: データ初期化方法の定義

テスト実行前のデータ初期化方法を定義する:

```markdown
## データ初期化

### 方針
- テスト実行前に初期状態にリセットする
- テスト間の独立性を確保する

### 初期化手順
1. ...
2. ...

### 現行環境での初期化
- 方法: ...
- スクリプト: ...

### 移行後環境での初期化
- 方法: ...
- スクリプト: ...
```

### Step 5: テスト間独立性の確保

テストが互いに影響しないようにする方針を定義する:

- テスト実行順序に依存しない設計
- テストごとのデータ分離方針（専用データ / 共有データ）
- テスト後のクリーンアップ方針

### Step 6: 後片付け方法の定義

```markdown
## 後片付け

### 方針
- テスト実行後にデータを削除/リセットする or しない
- 理由: ...

### 後片付け手順
1. ...
```

### Step 7: テストデータのバリエーション

各機能IDについて、テストに必要なデータバリエーションを定義する:

```markdown
## テストデータバリエーション

### F-XX-YY-ZZZ: <機能名>

| シナリオ | 必要データ | 期待結果 | 備考 |
|---|---|---|---|
| 正常系 | ... | ... | |
| データ0件 | | 空一覧表示 | |
| 大量データ | 100件以上 | ページング動作 | |
| 権限なし | test-userでアクセス | 403 or 非表示 | |
```

### Step 8: spec/test-data.md の作成

上記のすべてを `spec/test-data.md` にまとめる。

### 作業ログの記録先

- **初回（骨格作成）**: 横断的関心事として `spec/worklog/cross-cutting.md` に記録する。
- **2回目以降（機能ID単位の差分更新）**: 機能ID単位の作業として `spec/worklog/F-XX-YY.md` に記録する。

### Step 9: 終了処理

1. spec/traceability/F-XX-YY.md のステータスを `test-data-designed` に更新する。
2. 作業ログを更新する（初回: spec/worklog/cross-cutting.md、2回目以降: spec/worklog/F-XX-YY.md）。
3. 成果サマリーを出力する。
4. 次skill（`09-gherkin-generation`）を報告する。

## 制約

- テストデータ方針を決めるskillであり、E2Eテスト本体は作成しない。
- DB直接投入、API投入、画面操作投入のどれを使うかを根拠付きで整理する。
- 機密情報や本番データをテストデータとして直接扱わない。
- テストデータに本番環境の個人情報を含めない。
- 他phaseの成果物変更・対象外機能IDの作業は行わない。
