## UC-018: 新しい製品をカタログに登録して調達対象として利用可能にする

### アクター
- 全ロール（ADMIN, MANAGER, BUYER, WAREHOUSE_STAFF, VIEWER）
  - 技術的負債: @RolesAllowed未設定のため全認証済みユーザーがアクセス可能

### 目的
新規製品の基本情報・価格/在庫情報・仕様・画像・ドキュメントを登録し、調達・在庫管理・価格管理で利用可能な状態にする。

### 事前条件
- ユーザーがシステムにログイン済みであること
- 製品登録画面（/products/new）にアクセスしていること
- 登録する製品のSKU・カテゴリ・メーカー・価格情報が決定していること

### 基本フロー
1. ユーザーが製品一覧画面で「新規登録」ボタンをクリックする
2. システムが5ステップウィザード形式の製品登録画面を表示する（ステップ1: 基本情報）
3. システムがGET /api/products/categories, GET /api/products/manufacturersでカテゴリ・メーカーの選択肢を取得する
4. ユーザーがステップ1で基本情報（製品名・SKU・説明・カテゴリ・メーカー・ステータス）を入力する
5. システムがSKU入力時にGET /api/products/check-skuで重複チェックを実行し、結果をインジケーター表示する
6. ユーザーが「次へ →」をクリックする。システムがバリデーションを実行し、合格時にステップ2に遷移する
7. ユーザーがステップ2で価格・在庫情報（単価・単位・最低発注数・リードタイム・重量・寸法）を入力する
8. ユーザーが「次へ →」をクリックし、ステップ3に遷移する
9. ユーザーがステップ3で仕様テーブル（動的キーバリュー行）と備考を入力する
10. ユーザーが「次へ →」をクリックし、ステップ4に遷移する
11. ユーザーがステップ4で製品画像（PNG/JPG）とドキュメント（PDF）を選択する
12. ユーザーが「次へ →」をクリックし、ステップ5（確認画面）に遷移する
13. システムが全入力内容を読み取り専用テーブルで表示する
14. ユーザーが内容を確認し「登録する」をクリックする
15. システムがPOST /api/productsで製品を登録する
16. システムが返却された製品IDを使ってPOST /api/products/{id}/images, POST /api/products/{id}/documentsで画像・ドキュメントをforkJoinで並列アップロードする
17. システムが製品一覧画面（/products）へ遷移する

### 代替フロー
- A1: 「← 前へ」で前のステップに戻る → 入力値は保持される（バリデーションなし）
- A2: ステップ3（仕様）・ステップ4（画像・資料）は全項目任意のため、未入力のままスキップ可能
- A3: ステップ2の価格・在庫情報はデフォルト値（単価=0, 単位=個, 最低発注数=1, リードタイム=7日）が設定済みのため変更なしで遷移可能

### 例外フロー
- E1: ステップ1で必須項目未入力のまま「次へ →」 → バリデーションエラーメッセージ表示、ステップ遷移しない
- E2: SKU重複 → SKUフィールドに重複エラー表示（リアルタイムチェック）
- E3: SKUパターン不一致 → パターンバリデーションエラー表示（技術的負債: FE/BE間でパターン不一致）
- E4: POST /api/products失敗 → エラーメッセージ表示、画面遷移しない
- E5: 製品登録成功後に画像/ドキュメントアップロード失敗 → 製品は登録済みだが添付ファイルが未登録の状態になる
- E6: 「キャンセル」ボタンクリック → 確認なしに製品一覧画面へ遷移（技術的負債: unsaved changesガード未実装）

### 事後条件
- 新しい製品がDBに登録されている
- 選択した画像・ドキュメントがアップロードされている
- 製品一覧画面に新しい製品が表示される
- 監査ログに製品登録の記録が残る

### 関連機能ID

- F-03-05-003（SKU重複チェック）

### 関連API
- GET /api/products/categories（カテゴリ一覧取得）
- GET /api/products/manufacturers（メーカー一覧取得）
- GET /api/products/check-sku?sku={sku}（SKU重複チェック）
- POST /api/products（製品登録）
- POST /api/products/{id}/images（画像アップロード）
- POST /api/products/{id}/documents（ドキュメントアップロード）

### 根拠
- 画面操作: /products/newで5ステップウィザードの全ステップを操作確認
- ソースコード: product-create.component.ts（currentStep, nextStep, prevStep, submit, forkJoin）
- ネットワーク: POST /api/products → 201 Created、POST /api/products/{id}/images、POST /api/products/{id}/documents を確認
- API仕様: spec/api/F-03-05.yml

- 確認画面で仕様・画像・ドキュメント情報が表示されるかどうか
- ステップインジケーターのクリックによる直接ジャンプの可否

## UC-019: 既存製品の情報を更新してカタログの正確性を維持する

### アクター
- 全ロール（ADMIN, MANAGER, BUYER, WAREHOUSE_STAFF, VIEWER）
  - 技術的負債: @RolesAllowed未設定のため全認証済みユーザーが編集可能

### 目的
既存製品の基本情報・価格/在庫情報・仕様・画像・ドキュメントを最新の状態に更新し、カタログの正確性を維持する。

### 事前条件
- ユーザーがシステムにログイン済みであること
- 更新対象の製品が存在すること
- 製品詳細画面（/products/:id）から「編集」ボタンをクリックして編集画面（/products/:id/edit）にアクセスしていること

### 基本フロー
1. ユーザーが製品詳細画面で「編集」ボタンをクリックする
2. システムがGET /api/products/{id}で既存データを取得し、GET /api/products/categories, GET /api/products/manufacturersでカテゴリ・メーカーの選択肢を取得する
3. システムが全セクション（基本情報・価格/在庫・仕様・備考・画像・ドキュメント）を1ページにまとめた編集フォームを表示する
4. システムがフォームの各項目に既存値をプリセットする（仕様テーブルには既存の仕様データが行として表示される）
5. ユーザーが必要な項目を修正する
6. ユーザーが「更新する」をクリックする
7. システムがフォームバリデーションを実行する
8. システムがPUT /api/products/{id}で製品データを更新する
9. システムが製品一覧画面へ遷移する

### 代替フロー
- A1: 画像の追加 → 「+ 画像を追加」で即座にPOST /api/products/{id}/imagesが実行される（「更新する」と独立）
- A2: 画像の削除 → 「削除」ボタンで即座にDELETE /api/products/{id}/images/{imageId}が実行される（「更新する」と独立）
- A3: ドキュメントの追加 → ドキュメントタイプを選択してPOST /api/products/{id}/documentsが即時実行される（「更新する」と独立）
- A4: ドキュメントの削除 → DELETE /api/products/{id}/documents/{docId}が即時実行される（「更新する」と独立）
- A5: 仕様テーブルの行追加/削除 → 「+ 仕様行を追加」で空行追加、×ボタンで行削除（最低1行は削除不可）
- A6: SKU変更時にリアルタイム重複チェック → 自身のSKUは重複対象から除外
- A7: ステータスを「廃番」に変更 → 編集画面では新規登録にない「廃番」ステータスが選択可能（4択）

### 例外フロー
- E1: 製品ID不正（404） → 一覧画面へリダイレクト
- E2: 必須項目未入力で「更新する」 → markAllAsTouched()でバリデーションエラー表示
- E3: SKU重複（他製品と同一SKU） → サーバーエラー
- E4: PUT /api/products/{id}失敗 → エラーメッセージ表示
- E5: 画像/ドキュメント削除API失敗 → エラーメッセージ表示
- E6: 画像/ドキュメントアップロード失敗 → エラーメッセージ表示
- E7: 「キャンセル」ボタンクリック → 確認なしに製品一覧画面へ遷移（技術的負債: unsaved changesガード未実装。画像/ドキュメントの追加/削除は即時実行済みのためキャンセルしても取り消されない）

### 事後条件
- 製品情報がDBで更新されている
- 画像・ドキュメントの追加/削除が反映されている
- 監査ログに製品更新の記録が残る

### 関連機能ID
- F-03-05-003（SKU重複チェック）
- F-03-05-005（仕様テーブル入力）
- F-03-05-009（編集フォーム）

### 関連API

### 根拠

- API仕様: spec/api/F-03-05.yml

### confidence
High

### 未確認事項

- 更新成功後のリダイレクト先（製品一覧 or 製品詳細）
- 更新成功時のトースト/バナーメッセージの有無
- 画像/ドキュメントの即時削除に確認ダイアログがあるかどうか
- プライマリ画像の変更/設定操作の有無
