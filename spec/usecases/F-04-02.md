### UC-022: サプライヤーの詳細情報を確認して取引判断の材料を得る

#### アクター
- 全ロール（ADMIN, MANAGER, BUYER, WAREHOUSE_STAFF, VIEWER）
- 主な利用者: BUYER, MANAGER（調達意思決定者）

#### 目的
特定のサプライヤーの基本情報・連絡先・取扱製品・契約状況・評価履歴・認証状態を横断的に確認し、取引継続の判断やコミュニケーション、発注先選定の材料を得る。

#### 事前条件
- ユーザーがシステムにログイン済みであること
- 対象サプライヤーが登録済みであること
- ユーザーがサプライヤー一覧画面（/suppliers）または直リンクから詳細画面にアクセスすること

#### 基本フロー
1. ユーザーがサプライヤー詳細画面（/suppliers/{id}）を開く
2. システムがGET /api/suppliers/{id}でサプライヤー基本情報を取得し、ヘッダ領域に会社名・ステータス・コード・評価を表示する
3. システムが同時にGET /api/suppliers/{id}/contacts, /products, /contracts, /ratings, /certificationsを呼び出し、各タブのデータをロードする
4. ユーザーが基本情報タブ（デフォルト表示）で会社情報テーブルと連絡先カードを確認する
5. ユーザーが必要に応じてタブを切り替え、製品・契約・評価・認証の情報を確認する
6. ユーザーが確認を終え、「← 一覧に戻る」ボタンでサプライヤー一覧に戻る

#### 代替フロー
- A1: 製品タブで取扱製品の詳細を確認 → 「詳細」ボタンで製品詳細画面（/products/{productId}）に遷移する
- A2: 編集が必要な場合 → 「編集」ボタンでサプライヤー編集画面（/suppliers/{id}/edit）に遷移する
- A3: 評価履歴タブで平均評価と個別評価を確認 → 品質/納期/価格/対応の4カテゴリの経時変化を把握する

#### 例外フロー
- E1: サプライヤーが存在しない → APIエラー（現在500、本来404が適切）
- E2: 各タブのデータが0件 → 空状態メッセージが表示される（例: 「取扱製品はありません」「契約はありません」）

#### 事後条件
- ユーザーがサプライヤーの基本情報・連絡先・取扱製品・契約・評価・認証の状態を把握している
- 必要に応じて関連画面（製品詳細・編集画面・一覧画面）に遷移している

#### 関連機能ID
- F-04-02-001（サプライヤーヘッダ情報）
- F-04-02-002（一覧に戻るボタン）
- F-04-02-003（編集ボタン）

- F-04-02-006（連絡先セクション）

#### 関連API

- GET /api/suppliers/{id}/contacts
- GET /api/suppliers/{id}/products
- GET /api/suppliers/{id}/contracts
- GET /api/suppliers/{id}/ratings
- GET /api/suppliers/{id}/certifications

#### 根拠
- 画面操作: SUP-001（大塚商会/ACTIVE）とSUP-012（テクノアルファ/SUSPENDED）で5タブの表示と空状態を確認
- スクリーンショット: screenshots/F-04/F-04-02-001, 007, 008, 009, 010

### UC-023: サプライヤーの取引関係を管理して契約・製品紐付け・認証を整備する

#### アクター
- 全ロール（ADMIN, MANAGER, BUYER, WAREHOUSE_STAFF, VIEWER）
- 主な利用者: BUYER（製品紐付け）, MANAGER（契約管理）, ADMIN（認証管理）

#### 目的
サプライヤーとの取引関係を構成する情報（取扱製品の紐付け、契約の作成・更新・終了、認証の登録・更新）を管理し、正確な取引基盤を維持する。

#### 事前条件
- ユーザーがシステムにログイン済みであること
- 対象サプライヤーの詳細画面（/suppliers/{id}）を表示していること

#### 基本フロー（契約管理）
1. ユーザーが契約タブ（activeTab === 2）を選択する
2. 現在の契約一覧がテーブルに表示される（契約番号・期間・ステータス・条件）
3. ユーザーが「+ 新規契約」ボタンをクリックし、契約モーダルを表示する
4. ユーザーが契約番号・契約名・開始日・終了日・ステータス・条件を入力する
5. ユーザーが保存し、POST /api/suppliers/{id}/contractsで新規契約が作成される
6. テーブルに新しい契約行が追加される

#### 代替フロー
- A1: 契約編集 → テーブルの「編集」ボタンクリック → 契約モーダルに既存値がセット → 更新保存（PUT）
- A2: 契約削除 → テーブルの「削除」ボタンクリック → 確認ダイアログ → 物理削除（DELETE）
- A3: 製品紐付け追加 → 製品タブで「+ 製品紐付け追加」ボタンクリック → モーダルで製品選択・単価・リードタイム入力 → POST
- A4: 製品紐付け削除 → 製品テーブルの「削除」ボタンクリック → 確認ダイアログ → 紐付け削除（DELETE）
- A5: 認証追加 → 認証タブで「+ 新規認証」ボタンクリック → 認証モーダルで種別・番号・発行日・有効期限・ステータス入力 → POST
- A6: 認証編集 → 認証カードの「編集」ボタンクリック → モーダルに既存値セット → 更新保存（PUT）
- A7: 認証削除 → 認証カードの「削除」ボタンクリック → 確認ダイアログ → 物理削除（DELETE）

#### 例外フロー
- E1: API操作失敗 → エラーメッセージ表示（作成失敗/更新失敗/削除失敗）
- E2: 所有権チェック不一致（契約/認証がこのサプライヤーに属さない） → 404エラー
- E3: 製品が見つからない場合 → 400エラー

#### 事後条件
- サプライヤーの取引関連情報（契約・製品紐付け・認証）が最新の状態に更新されている
- 画面のテーブル/カードリストが更新内容を反映している

#### 関連機能ID
- F-04-02-007（製品タブ: 紐付け追加・削除）
- F-04-02-008（契約タブ: CRUD）
- F-04-02-010（認証タブ: CRUD）

#### 関連API
- GET /api/suppliers/{id}/products
- POST /api/suppliers/{id}/products
- DELETE /api/suppliers/{id}/products/{spId}
- GET /api/suppliers/{id}/contracts
- POST /api/suppliers/{id}/contracts
- PUT /api/suppliers/{id}/contracts/{contractId}
- DELETE /api/suppliers/{id}/contracts/{contractId}
- GET /api/suppliers/{id}/certifications
- POST /api/suppliers/{id}/certifications
- PUT /api/suppliers/{id}/certifications/{certId}
- DELETE /api/suppliers/{id}/certifications/{certId}

#### 根拠
- 画面操作: 契約タブでCNT-2024-001の表示、認証タブでISO 9001:2015等3件の表示を確認
- スクリーンショット: screenshots/F-04/F-04-02-008_contracts-tab.png, screenshots/F-04/F-04-02-007_products-tab.png, screenshots/F-04/F-04-02-010_certifications-tab.png
- ソースコード: supplier-detail.component.ts（saveContract, deleteContract, saveProduct, deleteProduct, saveCertification, deleteCertification）
- ソースコード: SupplierResource.java（契約/製品/認証の各CRUDエンドポイント）

#### confidence
High

### UC-024: サプライヤーを定期評価してパフォーマンスを経時的に記録する

#### アクター
- 全ロール（ADMIN, MANAGER, BUYER, WAREHOUSE_STAFF, VIEWER）
- 主な利用者: BUYER, MANAGER（サプライヤー評価担当者）

#### 目的
サプライヤーのパフォーマンスを品質・納期・価格・対応の4カテゴリで定量的に評価し、評価履歴を蓄積して取引先選定や改善要求の根拠とする。

#### 事前条件
- ユーザーがシステムにログイン済みであること
- 対象サプライヤーの詳細画面（/suppliers/{id}）を表示していること
- ユーザーが評価対象期間のサプライヤーとの取引実績を把握していること

#### 基本フロー
1. ユーザーが評価履歴タブ（activeTab === 3）を選択する
2. システムが既存の評価一覧を表示する（平均評価サマリー + 個別評価エントリリスト）
3. ユーザーが「+ 新規評価」ボタンをクリックし、評価モーダルを表示する
4. ユーザーが4カテゴリのスライダーで評価値を入力する（品質/納期/価格/対応、各1.0〜5.0、step=0.5）
5. ユーザーがコメントを入力する（任意）
6. ユーザーが保存し、POST /api/suppliers/{id}/rateで評価が追加される
7. overallScoreが4カテゴリの平均で自動計算され、ratingPeriodが現在の四半期で自動設定される
8. 評価リストが再ロードされ、新しいエントリと更新された平均評価が表示される

#### 代替フロー
- A1: 過去の評価を参照して評価値を決定 → 評価エントリリストで過去のスコア推移を確認してから新規評価を入力

#### 例外フロー
- E1: 評価追加失敗 → エラーメッセージ表示
- E2: 評価履歴が0件 → 「評価履歴はありません」空状態メッセージ表示、「+ 新規評価」ボタンは利用可能

#### 事後条件
- 新しい評価が評価履歴に追加されている
- 平均評価が再計算されている
- サプライヤーの全体ratingが再計算されている（ただしヘッダのrating表示には反映されない: 技術的負債）

#### 関連機能ID
- F-04-02-009（評価履歴タブ: 表示 + 新規評価追加）

#### 関連API
- GET /api/suppliers/{id}/ratings
- POST /api/suppliers/{id}/rate

#### 根拠
- 画面操作: SUP-001の評価履歴タブで4件の評価・平均4.5の表示を確認
- スクリーンショット: screenshots/F-04/F-04-02-009_ratings-tab.png
- ソースコード: supplier-detail.component.ts（loadRatings, submitRating）、SupplierResource.java（rateSupplier）、SupplierServiceBean.java（addRating）
- API仕様: POST /api/suppliers/{id}/rate のリクエスト/レスポンス構造

#### confidence
High

#### 未確認事項
- 評価の編集・削除機能の有無（UIにボタンが存在しないため、追加のみの可能性）
- evaluatedByフィールドにユーザーID番号が表示される問題の移行後対応方針

### UC-025: 不要になったサプライヤーを取引先から除外する

#### アクター
- 全ロール（ADMIN, MANAGER, BUYER, WAREHOUSE_STAFF, VIEWER）
- 主な利用者: ADMIN, MANAGER（サプライヤー管理権限者）

#### 目的
取引を停止するサプライヤーをシステムから論理的に削除（無効化）し、今後の調達候補から除外する。

#### 事前条件
- ユーザーがシステムにログイン済みであること
- 対象サプライヤーの詳細画面（/suppliers/{id}）を表示していること
- 対象サプライヤーとの取引を停止する判断がされていること

#### 基本フロー
1. ユーザーがヘッダ領域の「削除」ボタンをクリックする
2. システムが確認ダイアログを表示する（タイトル: 「サプライヤーの削除」、メッセージ: 「このサプライヤーを削除してもよろしいですか？関連する契約情報も削除されます。」）
3. ユーザーが「削除する」ボタンをクリックする
4. システムがDELETE /api/suppliers/{id}を呼び出す（論理削除: ステータスをINACTIVEに変更）
5. システムがサプライヤー一覧画面（/suppliers）に遷移する

#### 代替フロー
- A1: 削除キャンセル → 確認ダイアログの「キャンセル」ボタンクリック → ダイアログが閉じて詳細画面に戻る

#### 例外フロー
- E1: DELETE API失敗 → 「サプライヤーの削除に失敗しました。」エラーメッセージが表示される
- E2: 存在しないサプライヤーIDの場合 → 500エラー（本来は404が適切）

#### 事後条件
- サプライヤーのステータスがINACTIVEに変更されている（物理削除ではない）
- ユーザーがサプライヤー一覧画面に戻っている

#### 関連機能ID
- F-04-02-004（削除ボタン）

#### 関連API
- DELETE /api/suppliers/{id}

#### 根拠
- 画面操作: 削除ボタンクリック後の確認ダイアログ表示を確認
- スクリーンショット: screenshots/F-04/F-04-02-004_delete-confirm.png
- ソースコード: supplier-detail.component.ts（deleteSupplier）、SupplierResource.java（deleteSupplier）、SupplierServiceBean.java（deactivateSupplier: ステータスをINACTIVEに変更）
- Network通信: DELETE /api/suppliers/9999 → 500エラー確認

#### confidence
High

#### 未確認事項
- 論理削除後のサプライヤーが一覧画面にどう表示されるか（INACTIVEフィルタで表示される可能性）
- 削除時に関連契約が本当に削除されるのか、それとも確認ダイアログのメッセージのみで実際は論理削除のみか
