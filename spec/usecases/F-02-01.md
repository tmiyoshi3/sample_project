## UC-004: 業務状況を俯瞰して当日の優先対応を判断する

### アクター
- 全ロール（ADMIN, MANAGER, BUYER, WAREHOUSE_STAFF, VIEWER）

### 目的
ログイン後にダッシュボード画面で調達・在庫・承認・予算の概況を一目で把握し、当日の優先アクション（在庫補充、承認処理、発注確認など）を判断する。

### 事前条件
- ユーザーがシステムにログイン済みである（UC-001完了済み）
- Keycloakセッションが有効である

### 基本フロー
1. ユーザーがダッシュボード画面（/dashboard）にアクセスする（ログイン後に自動リダイレクト、またはサイドナビゲーションから遷移）
2. ローディングスピナーが表示され、7つのAPIが呼び出される（直列実行）
3. サマリーカード4枚が表示される:
   - 発注件数: 全発注書の合計件数と保留中・承認済みの内訳
   - 在庫アラート: 在庫不足（発注点以下）の製品数
   - 承認待ち: 承認が必要なタスク数（発注書承認待ち + 購買依頼承認待ち）
   - 予算消化率: 当年度の予算消化率（パーセント + プログレスバー + 金額内訳）
4. チャートエリア2セクションが表示される:
   - 月別発注金額推移: 過去12か月の月別発注金額（棒グラフ）
   - カテゴリ別支出割合: カテゴリごとの支出金額と構成比（水平バー + パーセント + 金額）
5. 最近の発注テーブル（上位10件、発注日降順）が表示される
6. 在庫アラートパネル（在庫不足製品リスト）が表示される
7. ユーザーが各サマリー値を確認し、注意が必要な項目（在庫アラート > 0、承認待ち > 0、予算消化率 > 80%など）を識別する

### 代替フロー
- A1: 在庫不足がない場合 → 在庫アラートパネルに「在庫アラートはありません」が表示される
- A2: 発注データがない場合 → 最近の発注テーブルに「最近の発注はありません」が表示される
- A3: 予算データが未登録の場合 → 予算消化率は0%、金額は¥0/¥0で表示される
- A4: 月別支出データがない場合 → 棒グラフは表示されず「※ グラフはプレースホルダーです」注記のみ表示される

### 例外フロー
- E1: API取得エラー → 画面上部にエラーバナー「データの取得に失敗しました。再度お試しください。」と「再試行」ボタンが表示される。各カードは0/空で表示される
- E2: セッション期限切れ → AuthInterceptorがBearerトークンの自動更新を試行。失敗した場合、ErrorInterceptorが401を検知しKeycloakログインを再トリガーする

### 事後条件
- ユーザーが調達・在庫・承認・予算の現在状況を把握している
- 優先対応が必要な項目（在庫不足、承認待ち、予算超過リスク）を識別している
- 必要に応じて詳細画面への遷移を判断できる状態になっている

### 関連機能ID
- F-02-01-001（発注件数カード）
- F-02-01-002（在庫アラートカード）
- F-02-01-003（承認待ちカード）
- F-02-01-004（予算消化率カード）
- F-02-01-005（月別発注金額推移グラフ）
- F-02-01-006（カテゴリ別支出割合）
- F-02-01-007（最近の発注テーブル）
- F-02-01-009（在庫アラートテーブル）

### 関連API
- GET /api/dashboard/summary
- GET /api/dashboard/spending-trend?months=12
- GET /api/dashboard/category-spending
- GET /api/purchase-orders?page=0&size=100
- GET /api/inventory/alerts/low-stock
- GET /api/budgets?fiscalYear=2026
- GET /api/requisitions?page=0&size=100&status=PENDING_APPROVAL

### 根拠
- 画面操作: ダッシュボード画面の全セクション表示確認（サマリーカード4枚、チャート2セクション、テーブル2セクション）
- Network通信: 7つのAPIリクエスト/レスポンスを確認
- ソースコード: dashboard-main.component.ts（loadDashboardSummary: 7つのAPIを直列呼び出し）、DashboardResource.java（6エンドポイント）
- スクリーンショット: screenshots/F-02/F-02-01_dashboard_initial_phase3.png

### confidence
High

### 未確認事項
- 発注書が100件を超えた場合の発注件数カードの正確性（page=0&size=100で取得）
- 予算消化率 > 80%/95%時のプログレスバーの実際の色変化（テスト環境に予算データなし）
- 月別発注金額推移グラフの実際の棒グラフ表示（テスト環境にデータなし）
- ロール別の表示差異の有無（現状は全ロールに同一データが表示される）
- 承認待ちカウントの二重加算の可能性（summary.pendingApprovalsにrequisitionsの件数が含まれているかどうか）
- 会計年度2026がハードコードされている問題

## UC-005: 在庫不足を検知して補充対応を開始する

### アクター
- 全ロール（ADMIN, MANAGER, BUYER, WAREHOUSE_STAFF, VIEWER）
- 主にBUYER, WAREHOUSE_STAFF, MANAGERが実際のアクションを取る

### 目的
ダッシュボードの在庫アラートから在庫不足の製品を検知し、該当製品の詳細確認または在庫管理画面への遷移を通じて、補充発注の対応を開始する。

### 事前条件
- ユーザーがダッシュボード画面を表示している（UC-004の基本フロー完了後）
- 在庫不足（発注点以下）の製品が存在する

### 基本フロー
1. ユーザーがダッシュボードの在庫アラートカード（F-02-01-002）で在庫不足製品数を確認する（has-alertクラスで強調表示）
2. ユーザーが在庫アラートパネル（F-02-01-009）で在庫不足製品のリストを確認する:
   - 製品名・SKU
   - 現在在庫数と最低在庫数
   - 在庫バー（critical: 最低在庫の50%以下 / warning: 最低在庫以下）
3. ユーザーが特定の在庫不足製品をクリックする
4. 製品詳細画面（/products/:id）に遷移し、製品情報を確認する
5. ユーザーが補充発注の必要性を判断する

### 代替フロー
- A1: 在庫アラートカードをクリックした場合 → 在庫管理画面（/inventory）に遷移し、全在庫状況を確認する
- A2: 「すべて表示 →」リンク（F-02-01-010）をクリックした場合 → 在庫管理画面（/inventory）に遷移する
- A3: 在庫不足製品がない場合 → 在庫アラートカードは「0」表示、在庫アラートパネルは「在庫アラートはありません」表示。ユーザーは対応不要と判断する

### 例外フロー
- E1: 在庫アラートAPI取得エラー → コンソールにエラー出力。カードは0表示、パネルは空状態表示。在庫不足の検知ができない

### 事後条件
- ユーザーが在庫不足の製品を特定している
- 製品詳細画面または在庫管理画面に遷移し、補充対応のアクションを開始できる状態になっている

### 関連機能ID
- F-02-01-002（在庫アラートカード）
- F-02-01-009（在庫アラートテーブル）
- F-02-01-010（すべて表示リンク（在庫））

### 関連API
- GET /api/dashboard/summary（lowStockItems）
- GET /api/inventory/alerts/low-stock

### 根拠
- 画面操作: 在庫アラートカード表示確認（0件）、在庫アラートパネル空状態表示確認
- ソースコード: dashboard-main.component.ts — navigateToInventory()（/inventory遷移）、navigateToProduct()（/products/:id遷移）
- ソースコード: dashboard-main.component.html:40-53（カードテンプレート）、174-211（アラートリストテンプレート）
- 推測: 在庫不足時のリスト表示・在庫バーの動作はソースコードからの推定（テスト環境に該当データなし）

### confidence
Medium

### 未確認事項
- 在庫不足製品が存在する場合のリスト表示状態（テスト環境に該当データなし）
- 在庫バーのcritical/warningのCSS定義（実際の色・スタイル）
- アラートリストの表示上限（ページングやスクロールの有無）
- VIEWERロールのユーザーが在庫不足を検知しても発注権限がない場合の業務フロー

## UC-006: 承認待ちタスクを確認して承認処理に着手する

### アクター
- 全ロール（ADMIN, MANAGER, BUYER, WAREHOUSE_STAFF, VIEWER）
- 主にMANAGER, ADMINが承認処理を実行する

### 目的
ダッシュボードの承認待ちカードから未処理の承認タスク数を把握し、マイタスク画面に遷移して承認処理を開始する。承認遅延を防止し、調達プロセスを円滑に進める。

### 事前条件
- ユーザーがダッシュボード画面を表示している（UC-004の基本フロー完了後）
- 承認待ちの発注書または購買依頼が存在する

### 基本フロー
1. ユーザーがダッシュボードの承認待ちカード（F-02-01-003）で承認待ちタスク数を確認する（has-pendingクラスで強調表示）
2. 承認待ちカウントの内訳を把握する:
   - summary.pendingApprovals: 発注書の承認待ち数
   - requisitions(PENDING_APPROVAL).totalElements: 購買依頼の承認待ち数
3. ユーザーが承認待ちカードをクリックする
4. マイタスク画面（/dashboard/tasks）に遷移する
5. ユーザーが承認待ちタスクの一覧を確認し、承認・却下の処理を行う

### 代替フロー
- A1: 承認待ちタスクがない場合 → 承認待ちカードは「0」表示（has-pendingクラスなし）。ユーザーは承認対応不要と判断する

### 例外フロー
- E1: summary API取得エラー → コンソールにエラー出力。承認待ちカードは0表示。承認待ちタスクの検知ができない
- E2: マイタスク画面の内容が不明 → F-02-02-001は画面未到達のため、マイタスク画面の具体的な操作フローは未確認

### 事後条件
- ユーザーが承認待ちタスクの存在と件数を把握している
- マイタスク画面に遷移し、承認処理に着手できる状態になっている

### 関連機能ID
- F-02-01-003（承認待ちカード）
- F-02-02-001（マイタスク画面）

### 関連API
- GET /api/dashboard/summary（pendingApprovals）
- GET /api/requisitions?page=0&size=100&status=PENDING_APPROVAL
- GET /api/dashboard/pending-tasks（マイタスク画面用、画面未到達のため未確認）

### 根拠
- 画面操作: 承認待ちカード表示確認（4件）
- Network通信: GET /api/dashboard/summary → pendingApprovals:4、GET /api/requisitions?status=PENDING_APPROVAL → totalElements:0
- ソースコード: dashboard-main.component.ts — navigateToTasks()（/dashboard/tasks遷移）、L152-154（pendingApprovals合算ロジック）
- ソースコード: MyTasksComponent（app/features/dashboard/my-tasks/）の存在をソース構造解析で確認
- 推測: マイタスク画面の操作フローはソースコードの存在確認のみ（画面未到達）

### confidence
Medium

### 未確認事項
- マイタスク画面（/dashboard/tasks）の具体的な表示内容・操作（画面未到達）
- 承認待ちカウントの二重加算の可能性（summary.pendingApprovalsに購買依頼の件数が含まれているかどうか）
- 承認権限を持たないロール（BUYER, WAREHOUSE_STAFF, VIEWER）がカードをクリックした場合の挙動
- 承認待ちタスクが自分に割り当てられたものか、全体のものかの区別（現状は全体カウント）

## UC-007: 最近の発注状況を確認して個別発注を追跡する

### アクター
- 全ロール（ADMIN, MANAGER, BUYER, WAREHOUSE_STAFF, VIEWER）

### 目的
ダッシュボードの最近の発注テーブルから直近の発注書のステータスを確認し、必要に応じて個別の発注書詳細画面に遷移して進捗を追跡する。また、全発注書一覧画面に遷移して詳細な検索・フィルタリングを行う。

### 事前条件
- ユーザーがダッシュボード画面を表示している（UC-004の基本フロー完了後）
- 発注書データが存在する

### 基本フロー
1. ユーザーがダッシュボードの発注件数カード（F-02-01-001）で全発注書の合計件数と保留中・承認済みの内訳を確認する
2. ユーザーが最近の発注テーブル（F-02-01-007）で上位10件の発注書を確認する:
   - 発注番号（例: PO-2025-000006）
   - サプライヤー名
   - 発注日（YYYY年MM月DD日形式）
   - 金額（¥カンマ区切り）
   - ステータス（下書き/申請中/承認済み/発注済み/却下/キャンセル等）
3. ユーザーが特定の発注書の行をクリックする
4. 発注書詳細画面（/procurement/orders/:id）に遷移し、発注の詳細を確認する

### 代替フロー
- A1: 全発注書一覧を確認したい場合 → 「すべて表示 →」リンク（F-02-01-008）をクリックし、発注書一覧画面（/procurement/orders）に遷移する
- A2: 発注データがない場合 → 発注件数カードは「0」表示、テーブルは「最近の発注はありません」表示。ユーザーは発注状況の追跡が不要と判断する

### 例外フロー
- E1: 発注書一覧API取得エラー → エラーバナー表示。発注件数カードは0、テーブルは空状態表示

### 事後条件
- ユーザーが直近の発注状況（件数・ステータス）を把握している
- 必要に応じて発注書詳細画面に遷移し、個別発注の進捗を追跡できる状態になっている

### 関連機能ID
- F-02-01-001（発注件数カード）
- F-02-01-007（最近の発注テーブル）
- F-02-01-008（すべて表示リンク（発注））

### 関連API
- GET /api/purchase-orders?page=0&size=100

### 根拠
- 画面操作: 発注件数カード表示確認（合計20件、保留中0件、承認済み2件）、最近の発注テーブル10件表示確認
- Network通信: GET /api/purchase-orders?page=0&size=100 → 20件のレスポンス確認
- ソースコード: dashboard-main.component.ts — calculateOrderStats()（ステータス別集計）、L136-138（発注日降順ソート・上位10件スライス）、navigateToOrder()（/procurement/orders/:id遷移）
- ソースコード: dashboard-main.component.html:132-171（テーブルテンプレート）、L137（「すべて表示 →」リンク）

### confidence
High

### 未確認事項
- 発注書が100件を超えた場合に最新10件の抽出が正しく行われるか（page=0&size=100で取得しているため、101件目以降は反映されない）
- ステータスラベルの全パターンと対応CSSクラスの完全なマッピング
- 発注書詳細画面（F-05-06）の内容（別大分類のため本ユースケースのスコープ外）
