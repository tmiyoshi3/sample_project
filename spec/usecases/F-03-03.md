## UC-012: カテゴリの階層構造を確認して分類方針を検討する

### アクター
- 全ロール（ADMIN, MANAGER, BUYER, WAREHOUSE_STAFF, VIEWER）

### 目的
現在の製品分類体系（階層構造・各カテゴリの製品数）を確認し、カテゴリの追加・統廃合・再分類の判断材料を得る。

### 事前条件
- ユーザーがシステムにログイン済みであること
- カテゴリ管理画面（/products/categories）にアクセスしていること

### 基本フロー
1. ユーザーがカテゴリ管理画面を開く
2. システムがGET /api/products/categoriesでカテゴリ一覧を取得する
3. システムがparentIdをもとにカテゴリツリーを構築し、全ノードを展開状態で表示する
4. ユーザーが各カテゴリの名前・製品数（直接属する製品のみ）を確認する
5. ユーザーが関心のある部分を展開/折りたたみして階層を確認する

### 代替フロー
- A1: カテゴリが0件の場合 → 空状態メッセージ「カテゴリが登録されていません」を表示
- A2: 特定の親カテゴリの子カテゴリのみ確認したい場合 → 他のカテゴリを折りたたんで対象のみ展開

### 例外フロー
- E1: API呼び出し失敗 → エラーバナー「カテゴリの取得に失敗しました。」を表示

### 事後条件
- ユーザーがカテゴリの階層構造と各カテゴリの製品数を把握している

### 関連機能ID
- F-03-03-001（カテゴリツリー）
- F-03-03-003（カテゴリ折りたたみ）
- F-03-03-008（製品数表示）

### 関連API
- GET /api/products/categories

### 根拠
- 画面操作: カテゴリ管理画面にアクセスし、ツリー表示・折りたたみ・製品数の表示を確認
- ソースコード: category-management.component.ts（loadCategories, buildTree, toggleExpand）、CategoryResource.java（listCategories）
- 推測: なし

### confidence
High

### 未確認事項
- productCountは直接属する製品数のみで子カテゴリの製品数を含まない。累計表示が必要かは業務要件として未確認

## UC-013: カテゴリを追加・編集して製品分類体系を整備する

### アクター
- ADMIN, MANAGER（本来の想定。現行は@RolesAllowed未設定のため全ロールが操作可能）

### 目的
新規カテゴリの作成、既存カテゴリの名前・説明・親カテゴリの変更、表示順の調整を行い、製品分類体系を維持・改善する。

### 事前条件
- ユーザーがシステムにログイン済みであること
- カテゴリ管理画面（/products/categories）にアクセスしていること
- カテゴリツリーが読み込まれていること（UC-012の基本フロー完了後）

### 基本フロー（ルートカテゴリ作成）
1. ユーザーが「+ 新規カテゴリ」ボタンをクリックする
2. システムが右側に「カテゴリ作成」フォームを表示する（親カテゴリ=ルート）
3. ユーザーがカテゴリ名（必須, 最大100文字）と説明（任意, 最大500文字）を入力する
4. ユーザーが「保存」ボタンをクリックする
5. システムがPOST /api/products/categoriesを呼び出す
6. システムが成功バナー「カテゴリを作成しました。」を表示し、カテゴリツリーを再読込する

### 代替フロー
- A1: 子カテゴリとして作成する場合 → 対象カテゴリの「+」ボタンをクリック → フォームの親カテゴリに対象カテゴリがプリセットされた状態で表示 → 以降は基本フロー3〜6と同じ
- A2: 「+ 新規カテゴリ」ボタンからフォーム表示後に親カテゴリをセレクトボックスで変更 → 任意の既存カテゴリの子として作成可能
- A3: 既存カテゴリを編集する場合 → 対象カテゴリの「✎」ボタンをクリック → 「カテゴリ編集」フォーム表示（既存値プリフィル）→ 値変更 → 保存 → PUT /api/products/categories/{id} → 成功バナー「カテゴリを更新しました。」→ ツリー再読込
- A4: カテゴリの表示順を変更する場合 → 対象カテゴリの「↑」/「↓」ボタンをクリック → 同一階層内で表示順が変わる（クライアントサイドのみ、非永続化）
- A5: 操作をキャンセルする場合 → 「キャンセル」ボタンをクリック → フォームが閉じる

### 例外フロー
- E1: カテゴリ名未入力で保存 → バリデーションエラー「カテゴリ名は必須です」を表示。APIは呼ばれない
- E2: API呼び出し失敗（作成時）→ エラーバナー「カテゴリの作成に失敗しました。」
- E3: API呼び出し失敗（更新時）→ エラーバナー「カテゴリの更新に失敗しました。」

### 事後条件
- カテゴリが作成/更新されている（A4の並べ替えを除く）
- カテゴリツリーが最新状態に更新されている

### 関連機能ID
- F-03-03-002（新規カテゴリボタン）
- F-03-03-004（上下移動ボタン）
- F-03-03-005（子カテゴリ追加ボタン）
- F-03-03-006（カテゴリ編集ボタン）

### 関連API
- POST /api/products/categories
- PUT /api/products/categories/{id}

### 根拠
- 画面操作: 新規カテゴリ作成フォーム・子カテゴリ追加・編集フォームの動作を確認
- ソースコード: category-management.component.ts（startCreate, startEdit, saveCategory, moveUp, moveDown）、ProductResource.java（createCategory, updateCategory）
- 推測: なし

### confidence
High

### 未確認事項
- 循環参照防止がFE/BEともに未実装（自身を親カテゴリに設定可能）
- FE/BE間のmaxLength不一致（FE: name=100/description=500, DB: name=200/description=1000）
- 上下移動の並べ替え結果がサーバーに永続化されない（リロードで元に戻る）。移行時に永続化が必要か要判断
- BE側バリデーションなし（FE側のReactive Formsバリデーションのみ）

## UC-014: 不要なカテゴリを削除してカタログ分類を整理する

### アクター
- ADMIN（本来の想定。現行は@RolesAllowed未設定のため全ロールが操作可能）

### 目的
不要になったカテゴリを削除して、製品カタログの分類体系を整理する。

### 事前条件
- ユーザーがシステムにログイン済みであること
- カテゴリ管理画面（/products/categories）にアクセスしていること
- 削除対象カテゴリに子カテゴリが存在しないこと
- 削除対象カテゴリに製品が紐付いていないこと

### 基本フロー
1. ユーザーが削除対象カテゴリの「✕」ボタンをクリックする
2. システムが確認ダイアログ「「{カテゴリ名}」を削除してもよろしいですか？」を表示する
3. ユーザーが「削除する」ボタンをクリックする
4. システムがDELETE /api/products/categories/{id}を呼び出す
5. システムが成功バナー「カテゴリを削除しました。」を表示し、カテゴリツリーを再読込する

### 代替フロー
- A1: 削除をキャンセルする場合 → 確認ダイアログの「キャンセル」ボタンをクリック → ダイアログが閉じる

### 例外フロー
- E1: 子カテゴリが存在する場合 → 409 Conflict → エラーバナー「子カテゴリが存在するため削除できません。」
- E2: 製品が紐付いている場合 → 409 Conflict → エラーバナー「このカテゴリに紐付く製品が存在するため削除できません。」
- E3: カテゴリが見つからない場合 → 404 Not Found → エラーバナー「カテゴリが見つかりません。」
- E4: その他のAPI失敗 → エラーバナー「カテゴリの削除に失敗しました。」

### 事後条件
- カテゴリが物理削除されている
- カテゴリツリーから対象カテゴリが除外されている

### 関連機能ID
- F-03-03-007（カテゴリ削除ボタン）

### 関連API
- DELETE /api/products/categories/{id}

### 根拠
- 画面操作: 削除確認ダイアログの表示を確認
- ソースコード: category-management.component.ts（confirmDelete, deleteCategory, cancelDelete）、ProductResource.java（deleteCategory — 子カテゴリ・製品チェック付き物理削除）
- 推測: なし

### confidence
High

### 未確認事項
- 削除操作のロール制限が未実装（@RolesAllowed未設定。全ロールが削除可能）
