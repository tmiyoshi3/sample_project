## UC-008: 製品カタログから目的の製品を検索・絞り込みして情報を確認する

### アクター
- 全ロール（ADMIN, MANAGER, BUYER, WAREHOUSE_STAFF, VIEWER）
- 主にBUYER（発注対象製品の確認）、WAREHOUSE_STAFF（在庫状況の確認）、MANAGER（カタログ全体の把握）が利用

### 目的
キーワード検索・カテゴリ/メーカー/ステータスフィルタ・ソート・ページングを組み合わせて、目的の製品（または条件に合致する製品群）を見つけ、価格・在庫数・ステータスなどの情報を確認する。必要に応じて製品詳細画面に遷移して仕様・サプライヤー・変更履歴などの詳細を確認する。

### 事前条件
- ユーザーがシステムにログイン済みである（UC-001完了済み）
- 製品カタログにデータが登録されている

### 基本フロー
1. ユーザーがサイドナビゲーション（F-12-02-002）から「製品カタログ」を選択し、製品一覧画面（/products）にアクセスする
2. 画面が初期表示され、以下が読み込まれる:
   - GET /api/products?page=0&size=20&sort=name,asc → 製品一覧（20件/ページ、製品名昇順）
   - GET /api/products/categories → カテゴリフィルタの選択肢（20カテゴリ）
   - GET /api/products?page=0&size=1000 → メーカーフィルタの選択肢抽出用（全製品からユニーク抽出）
3. 製品一覧テーブル（F-03-01-005）に7列（SKU・製品名・カテゴリ・メーカー・単価・ステータス・在庫数）が表示される
4. ユーザーが目的に応じてフィルタを適用する:
   - 検索ボックス（F-03-01-001）にキーワード入力 → 製品名・説明・SKUの部分一致検索
   - カテゴリフィルタ（F-03-01-002）で絞り込み → 20カテゴリから選択
   - メーカーフィルタ（F-03-01-003）で絞り込み → 15メーカーから選択
   - ステータスフィルタ（F-03-01-004）で絞り込み → 有効/無効/廃番/保留から選択
5. フィルタ適用のたびにGET /api/productsが呼び出され、テーブルが更新される。ページは1にリセットされる。「フィルターをリセット」リンクが表示される
6. ユーザーが列ヘッダクリック（F-03-01-006）でソート順を切り替える（SKU・製品名・カテゴリ・メーカー・単価・ステータスの6列が対象。昇順▲/降順▼トグル）
7. ユーザーがページナビゲーション（F-03-01-007）でページを切り替え、全件を確認する（20件/ページ、«‹1-5›»ボタン）
8. ユーザーが目的の製品行をクリック（F-03-01-012）し、製品詳細画面（/products/:id）に遷移して詳細情報を確認する

### 代替フロー
- A1: フィルタ条件を複数組み合わせる場合 → 検索キーワード・カテゴリ・メーカー・ステータスの各フィルタはAND条件で同時適用できる。例: カテゴリ「ノートPC」× メーカー「Dell」× ステータス「有効」
- A2: フィルタをリセットする場合 → 「フィルターをリセット」リンクをクリックし、全フィルタを解除して全製品一覧に戻る
- A3: 検索キーワードをクリアする場合 → 検索ボックスの×ボタンをクリックし、キーワードフィルタのみ解除する
- A4: 製品詳細に遷移せず一覧上で情報確認を完了する場合 → テーブルの7列（SKU・製品名・カテゴリ・メーカー・単価・ステータス・在庫数）で十分な情報が得られれば、行クリックなしで目的を達成する

### 例外フロー
- E1: 検索・フィルタ結果が0件の場合 → テーブルに空状態メッセージが表示される。ユーザーはフィルタ条件を変更するか、リセットして再検索する
- E2: API取得エラー → テーブルが空で表示される。ユーザーはページを再読み込みする
- E3: セッション期限切れ → AuthInterceptorがBearerトークンの自動更新を試行。失敗した場合、Keycloakログインにリダイレクトされる

### 事後条件
- ユーザーが目的の製品（群）の情報（価格・在庫数・ステータス・カテゴリ・メーカー）を確認している
- 必要に応じて製品詳細画面に遷移し、仕様・サプライヤー・変更履歴等の詳細を確認できる状態になっている

### 関連機能ID
- F-03-01-001（検索テキストボックス）
- F-03-01-002（カテゴリフィルタ）
- F-03-01-003（メーカーフィルタ）
- F-03-01-004（ステータスフィルタ）
- F-03-01-005（製品一覧テーブル）
- F-03-01-006（テーブルソート）
- F-03-01-007（ページング）
- F-03-01-012（製品行クリック）

### 関連API
- GET /api/products（keyword, categoryId, manufacturerId, status, page, size, sort パラメータ）
- GET /api/products/categories（カテゴリフィルタ選択肢取得）

### 根拠
- 画面操作: Chrome DevTools MCPで検索（Dell）、カテゴリフィルタ（ノートPC）、ステータスフィルタ（廃番）、ソート（SKU昇順）、ページ切替、行クリック遷移を確認
- スクリーンショット: 5枚（初期表示、Dell検索、ノートPCフィルタ、廃番フィルタ、SKUソート）
- Network通信: GET /api/products（各フィルタ条件）、GET /api/products/categories のレスポンス確認
- ソースコード: ProductListComponent（検索・フィルタ・ソート・ページング・行クリック処理）、ProductResource.java（サーバーサイドフィルタ・ページング）

### confidence
High

### 未確認事項
- sortパラメータがバックエンドで未処理（@QueryParam未定義）のため、実際のソート順は常にname ASC固定。フロントエンドのUI上ではソートが機能しているように見えるが、データ順は変わらない（技術的負債）
- 親カテゴリ選択時に子カテゴリの製品が含まれるか（ソースコード上は完全一致: p.category.id = :categoryId）
- ステータス表示名の不一致: FEではDISCONTINUED=「廃番」、BEのenum定義ではDISCONTINUED=「販売終了」
- DRAFT（下書き）ステータスがProductServiceBeanのインポート機能で使用されているが、フィルタ選択肢に含まれていない
- manufacturerIdフィルタがインメモリ実装のためデータ量増加時のパフォーマンス懸念
- 在庫数取得のN+1クエリ（20件/ページ → 20回の追加クエリ）
- ロール別の画面表示差異の有無（現状は@RolesAllowed未設定で全ロール同一）

## UC-009: 製品カタログデータをCSVエクスポートして外部ツールで活用する

### アクター
- 全ロール（ADMIN, MANAGER, BUYER, WAREHOUSE_STAFF, VIEWER）
- 主にADMIN, MANAGER（カタログ管理・分析）、BUYER（発注計画・価格比較）が利用

### 目的
製品カタログの一覧データをCSVファイルとしてダウンロードし、Excel等の外部ツールで在庫棚卸の突合、価格見直し分析、レポート作成などの業務に活用する。

### 事前条件
- ユーザーがシステムにログイン済みである（UC-001完了済み）
- 製品一覧画面（/products）を表示している
- エクスポート対象の製品データが存在する

### 基本フロー
1. ユーザーが製品一覧画面（/products）の「CSV出力」ボタン（F-03-01-010）をクリックする
2. フロントエンドがGET /api/products?page=0&size=10000を呼び出し、全製品データを取得する（実際にはMAX_PAGE_SIZE=100に制限され、最大100件）
3. クライアントサイドでCSV文字列を生成する:
   - ヘッダ行: SKU, 製品名, カテゴリ, メーカー, 単価, ステータス
   - UTF-8 BOM付き（Excelでの文字化け防止）
4. ブラウザのダウンロードダイアログが表示され、products_YYYY-MM-DD.csv がダウンロードされる
5. ユーザーがダウンロードしたCSVをExcel等で開き、分析・加工に利用する

### 代替フロー
- A1: バックエンドCSVエクスポートを利用する場合 → GET /api/products/export エンドポイントが存在するが、フロントエンドはこれを使用していない（技術的負債: 二重実装）。バックエンドのエクスポートはACTIVEステータスの製品のみが対象

### 例外フロー
- E1: APIエラー → CSVダウンロードが実行されない。エラーハンドリングの具体的な挙動は未確認
- E2: データが0件の場合 → ヘッダ行のみのCSVがダウンロードされる（推定）

### 事後条件
- ユーザーがCSVファイルをダウンロードし、外部ツールで利用可能な状態になっている

### 関連機能ID
- F-03-01-010（CSV出力ボタン）

### 関連API
- GET /api/products（page=0, size=10000 → 実質最大100件）
- GET /api/products/export（バックエンド側CSVエクスポート、フロントエンド未使用）

### 根拠
- 画面操作: Chrome DevTools MCPで「CSV出力」ボタン表示確認（btn-export, オレンジ枠）
- ソースコード: ProductListComponent.exportCsv()（クライアントサイドCSV生成: size=10000リクエスト→BOM付CSV→Blobダウンロード）
- ソースコード: ProductResource.java:994-1006（バックエンド側エクスポートエンドポイント）

### confidence
High

### 未確認事項
- 現在のフィルタ条件がCSV出力に反映されるか（ソースコード上はフィルタなしで全件取得）
- size=10000指定がMAX_PAGE_SIZE=100に制限されるため、100件超のデータでCSVが不完全になる
- バックエンド側GET /api/products/exportのCSVフォーマット詳細（ProductServiceBean.exportProductsToCsv未確認）
- ダウンロード中のローディング表示の有無
- CSVダウンロード失敗時のエラーメッセージ
