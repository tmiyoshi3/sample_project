## UC-015: バンドルの構成と価格を確認して調達・販売方針の判断材料を得る

### アクター
- 全ロール（ADMIN, MANAGER, BUYER, WAREHOUSE_STAFF, VIEWER）

### 目的
登録済みバンドル（セット商品）の構成製品・価格・割引率・ステータスを一覧で確認し、セット販売の採算性や構成の妥当性を判断する。

### 事前条件
- ユーザーがシステムにログイン済みであること
- バンドル管理画面（/products/bundles）にアクセスしていること

### 基本フロー
1. ユーザーがバンドル管理画面を開く
2. システムがGET /api/products/bundlesでバンドル一覧を取得する
3. システムがバンドルをカード形式で表示する（バンドル名の昇順）
4. ユーザーが各カードのバンドル名・ステータスバッジ（有効/無効/下書き）・説明を確認する
5. ユーザーが構成製品チップ（製品名×数量）で含まれる製品を確認する
6. ユーザーが定価合計（打ち消し線）・割引率（赤字）・バンドル価格（緑色）で価格情報を確認する

### 代替フロー
- A1: バンドルが0件の場合 → 空状態メッセージ「バンドルが登録されていません」を表示

### 例外フロー
- E1: API呼び出し失敗 → エラーバナー「バンドルの取得に失敗しました。」を表示

### 事後条件
- ユーザーがバンドルの構成・価格・ステータスを把握している

### 関連機能ID
- F-03-04-001（バンドル一覧）
- F-03-04-004（バンドル価格情報 — 一覧カード上の表示）
- F-03-04-007（バンドルステータス — 一覧カード上のバッジ表示）

### 関連API
- GET /api/products/bundles

### 根拠
- 画面操作: バンドル管理画面にアクセスし、3件のバンドルカード表示を確認
- ソースコード: bundle-management.component.ts（loadBundles）、ProductResource.java（listBundles）
- 推測: なし

### confidence
High

### 未確認事項
- 一覧カードのバンドル価格（bundlePrice）はDB保存値、定価合計（totalPrice）はAPI応答時にbundleItemsから再計算。製品単価が変わると乖離する可能性
- ページングなし（全件取得）。バンドル数が増えた場合のパフォーマンス

## UC-016: バンドルを作成・編集して製品のセット販売体系を整備する

### アクター
- ADMIN, MANAGER（本来の想定。現行は@RolesAllowed未設定のため全ロールが操作可能）

### 目的
新しいセット商品（バンドル）を作成したり、既存バンドルの構成製品・価格・ステータスを変更して、セット販売体系を維持・改善する。

### 事前条件
- ユーザーがシステムにログイン済みであること
- バンドル管理画面（/products/bundles）にアクセスしていること
- バンドル一覧が読み込まれていること（UC-015の基本フロー完了後）

### 基本フロー（新規バンドル作成）
1. ユーザーが「+ 新規バンドル作成」ボタンをクリックする
2. システムが作成フォームを表示し、一覧を非表示にする
3. ユーザーがバンドル名（必須, 最大200文字）・ステータス（有効/無効/下書き, デフォルト: 有効）・説明（任意, 最大1000文字）を入力する
4. ユーザーが製品検索欄にキーワードを入力する
5. システムが製品候補をクライアントサイドフィルタで検索し、ドロップダウンに最大10件表示する（先頭100件の製品が対象）
6. ユーザーが候補から製品をクリックして追加する（数量は1で初期化）
7. システムが構成製品テーブルに行を追加し、価格を自動再計算する（合計 = Σ(単価×数量)、割引額 = floor(合計×割引率/100)、バンドル価格 = 合計-割引額）
8. ユーザーが必要に応じて数量を変更、割引率を設定する
9. ユーザーが「保存」ボタンをクリックする
10. システムがPOST /api/products/bundlesを呼び出す（バンドルコードはBE側で自動生成: BDL-{timestamp}）
11. システムが成功バナー「バンドルを作成しました。」を表示し、フォームを閉じて一覧を再読み込みする

### 代替フロー
- A1: 既存バンドルを編集する場合 → 対象カードの「編集」ボタンをクリック → 編集フォーム表示（既存値プリセット: バンドル名・説明・ステータス・構成製品・割引率）→ 値変更 → 保存 → PUT /api/products/bundles/{id} → 成功バナー「バンドルを更新しました。」→ 一覧再読み込み
- A2: 構成製品の数量を変更する場合 → テーブル内の数量欄を直接変更 → 価格がリアルタイム再計算される
- A3: 構成製品を削除する場合 → テーブル行の×ボタンをクリック → 行が削除され価格が再計算される
- A4: 既にバンドル内にある製品を再度選択する場合 → 該当製品の数量が+1される
- A5: 操作をキャンセルする場合 → 「キャンセル」ボタンをクリック → フォームが閉じ一覧に復帰
- A6: ステータスを変更する場合 → フォームのセレクトボックスで有効/無効/下書きを選択（ステータス遷移に制約なし）

### 例外フロー
- E1: バンドル名未入力で保存 → markAllAsTouched()でフォーム無効化。submitが無視される（項目ごとのエラーメッセージ表示なし）
- E2: 構成製品0件で保存 → エラーバナー「バンドルには少なくとも1つの製品を追加してください。」
- E3: API呼び出し失敗（作成/更新時）→ エラーバナー「バンドルの保存に失敗しました。」
- E4: バンドルが見つからない場合（編集時、BE: 404）→ エラーバナー表示

### 事後条件
- バンドルが作成/更新されている
- バンドル一覧が最新状態に更新されている
- バンドル価格がBE側で計算・保存されている

### 関連機能ID
- F-03-04-002（新規バンドル作成ボタン）
- F-03-04-003（バンドル構成製品）
- F-03-04-004（バンドル価格情報 — フォーム内の計算・表示）
- F-03-04-005（バンドル編集ボタン）
- F-03-04-007（バンドルステータス — フォーム内のセレクト変更）

### 関連API
- POST /api/products/bundles
- PUT /api/products/bundles/{id}
- GET /api/products?page=0&size=100（構成製品候補取得）

### 根拠
- 画面操作: 新規作成フォーム・編集フォーム・製品検索・価格計算の動作を確認
- ソースコード: bundle-management.component.ts（startCreate, startEdit, saveBundle, searchProducts, addProductToBundle, removeProductFromBundle, recalculatePrices）、ProductResource.java（createBundle, updateBundle, addBundleItems）
- 推測: なし

### confidence
High

### 未確認事項
- バリデーションエラーの視覚的表示がない（フォームinvalid時はsubmitが無視されるだけ）
- 製品候補がpage=0&size=100の先頭100件に限定。101件目以降の製品はバンドルに追加できない
- FEとBEで価格計算ロジックが重複（FE: Math.floor、BE: RoundingMode.FLOOR）
- 編集時、BE側でbundleItemsを全削除→再追加（orphanRemoval）。構成製品の順序は保証されない（sort_orderカラム未使用）
- BE側にバリデーションなし。DB制約（NOT NULL、CHECK）に依存
- createdBy/updatedByが"system"固定（SecurityContextを使用していない）

## UC-017: 不要なバンドルを削除してカタログを整理する

### アクター
- ADMIN（本来の想定。現行は@RolesAllowed未設定のため全ロールが操作可能）

### 目的
不要になったバンドル（セット商品）をシステムから完全に除去して、カタログを整理する。

### 事前条件
- ユーザーがシステムにログイン済みであること
- バンドル管理画面（/products/bundles）にアクセスしていること
- バンドル一覧が読み込まれていること（UC-015の基本フロー完了後）

### 基本フロー
1. ユーザーが削除対象バンドルカードの「削除」ボタンをクリックする
2. システムが確認ダイアログ「「{バンドル名}」を削除してもよろしいですか？」を表示する
3. ユーザーが「削除する」ボタンをクリックする
4. システムがDELETE /api/products/bundles/{id}を呼び出す
5. システムが成功バナー「バンドルを削除しました。」を表示し、ダイアログを閉じて一覧を再読み込みする

### 代替フロー
- A1: 削除をキャンセルする場合 → 確認ダイアログの「キャンセル」ボタンをクリック → ダイアログが閉じる

### 例外フロー
- E1: バンドルが見つからない場合（BE: 404）→ エラーバナー表示
- E2: API呼び出し失敗 → エラーバナー「バンドルの削除に失敗しました。」

### 事後条件
- バンドルが物理削除されている（product_bundle_itemもCASCADE DELETEで自動削除）
- バンドル一覧から対象バンドルが除外されている

### 関連機能ID
- F-03-04-006（バンドル削除ボタン）

### 関連API
- DELETE /api/products/bundles/{id}

### 根拠
- 画面操作: 「会議室セット」の削除ボタンクリック→確認ダイアログ表示を確認
- ソースコード: bundle-management.component.ts（confirmDelete, deleteBundle, cancelDelete）、ProductResource.java（deleteBundle — 物理削除）
- DB定義: V002 product_bundle_item REFERENCES product_bundle(id) ON DELETE CASCADE
- 推測: なし

### confidence
High

### 未確認事項
- 物理削除（論理削除ではない）。製品詳細のDISCONTINUED（論理削除）とは異なる方針。移行後も物理削除を維持するか要判断
- 削除操作のロール制限が未実装（@RolesAllowed未設定。全ロールが削除可能）
