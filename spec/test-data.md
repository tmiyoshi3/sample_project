## データ投入方法

### 選定結果

| データ種別 | 投入方法 | 選定理由 |
|---|---|---|
| マスターデータ全般 | DB直接投入（Flyway） | 高速・確実。スキーマ変更時はFlywayマイグレーションで追従可能。テスト前提として安定的に投入する必要がある |
| トランザクションデータ全般 | DB直接投入（Flyway） | 複数テーブルにまたがる整合性のあるデータ（発注書+明細、サプライヤー+連絡先等）を一括投入する必要がある。API投入では依存関係の順序制御が複雑 |
| Keycloakユーザー | レルム設定ファイル | Keycloakの起動時にレルム設定（proquip-realm.json）からインポートされる。DB直接投入ではKeycloak内部スキーマに依存するため不適切 |
| テスト固有データ（テスト中に作成するデータ） | API投入 | テスト中にCRUD操作で作成するデータ。ビジネスロジックを通ることで整合性を確保 |

### 選定理由の詳細

1. **DB直接投入（Flyway）を基本とする理由:**
   - Flywayシードデータ（V020〜V025）が既に存在し、約1,000+レコードの整合性のあるデータが用意されている
   - テスト実行前にDB初期化 → Flywayマイグレーション実行で確実にベースラインを作れる
   - 移行後環境でも同一のFlywayマイグレーションを実行するため、現行・移行後で同じ前提データを保証できる
   - APIが未実装の段階でもテストデータを準備できる

2. **API投入を補助的に使用する理由:**
   - テスト中に動的に作成・変更するデータ（CRUD操作の検証対象）はAPI経由で作成する
   - 例: 新規発注書の作成テストでは、テスト中にPOST /api/purchase-ordersで発注書を作成する

3. **画面操作投入は不採用:**
   - テストデータ準備の段階で画面操作投入を使うと低速で脆弱
   - ただし、E2Eテスト本体の「操作」としてはPlaywrightで画面操作を行う（これはテストデータ準備ではなくテスト本体の責務）

## データ初期化

### 方針

- テスト実行前にDBを既知の初期状態にリセットする
- Flywayシードデータの状態をベースラインとする
- テスト間の独立性を確保するため、テストスイート開始前に毎回初期化する

### 初期化手順

1. 既存データの全削除（TRUNCATE CASCADE）
2. Flyway cleanを実行（マイグレーション履歴をリセット）
3. Flyway migrateを実行（V001〜V025のマイグレーションを再適用）
4. Keycloakレルムの初期化（Podman Compose再起動時にレルム設定からインポート）

### 現行環境での初期化

```bash
# 方法: Podman Compose でDB再起動 + Flyway マイグレーション
cd proquip
podman compose -f podman-compose.yml down
podman compose -f podman-compose.yml up -d --build

# Flywayマイグレーションは WildFly起動時に自動実行される
# Keycloakレルムは Keycloak起動時に自動インポートされる
```

### 移行後環境での初期化

```bash
# 方法: Podman Compose でDB再起動 + Flyway マイグレーション
cd migrated
podman compose down
podman compose up -d --build

# Flywayマイグレーションは Quarkus起動時に自動実行される（quarkus.flyway.migrate-at-start=true）
# Keycloakレルムは Keycloak起動時に自動インポートされる
```

### 高速初期化（テストスイート間）

テストスイート全体を再起動せずにデータだけをリセットする方法:


### 未確認事項

- 高速初期化スクリプトの動作検証（TRUNCATE CASCADEでFK制約が正しく処理されるか）

## テスト間独立性の確保

### 方針

1. **テスト実行順序に依存しない設計:**
   - 各テストシナリオはFlywayシードデータの状態を前提とする
   - テスト間でデータの依存関係を持たない（テストAで作成したデータをテストBで使わない）

2. **テストごとのデータ分離方針:**
   - **共有データ（読み取り専用）:** Flywayシードデータ（マスターデータ、既存の発注書・在庫等）はテスト間で共有する。これらのデータはテスト中に変更しない
   - **専用データ（テスト中に作成）:** テスト中にCRUD操作で作成するデータは、テスト固有のプレフィックスや識別子を使用して他テストと区別する
   - **変更するテストのグループ化:** データを変更するテスト（作成・更新・削除）は、同一テストファイル内でserial実行し、テストファイル単位でデータ初期化を行う

3. **テストデータの命名規則:**
   - テスト中に作成するデータには `TEST-` プレフィックスを付与する
   - 例: 発注番号 `TEST-PO-001`、製品SKU `TEST-PRD-001`
   - テスト後のクリーンアップで `TEST-` プレフィックスのデータを削除する

### Playwright設定での対応

```typescript
// playwright.config.ts
export default defineConfig({
  fullyParallel: false,  // シナリオの並列実行は行わない（データ競合回避）
  workers: 1,            // ワーカー1（シリアル実行）
  retries: 0,            // リトライなし（データ状態が変わるため）
});
```

### 未確認事項

- Playwright workers > 1 でのデータ分離方法（DBスキーマ分割やトランザクション分離が必要になる）
- テストデータのプレフィックス `TEST-` がシードデータのバリデーションルールに適合するか

## 後片付け

### 方針

- テスト実行後にデータを削除/リセットしない（テスト失敗時のデバッグのため）
- 次のテスト実行時の初期化で前回のデータをクリアする
- 理由: テスト失敗時にDBの状態を確認して原因調査を行うため、テスト後のクリーンアップは行わない

### テストスイートレベルの後片付け

```
[テスト開始前]
  ↓ データ初期化（Flyway baseline復元）
[テスト実行]
  ↓ テスト中に作成・変更されたデータはそのまま残る
[テスト終了後]
  ↓ 何もしない（デバッグ用にデータを残す）
[次のテスト開始前]
  ↓ データ初期化（前回のデータをクリアしてFlyway baseline復元）
```

### テストシナリオレベルの後片付け

- 読み取り専用テスト: 後片付け不要（データ変更なし）
- データ変更テスト: afterEach/afterAllで `TEST-` プレフィックスのデータを削除する（オプション）
  - ただし、テストスイートレベルの初期化で十分な場合はシナリオレベルの後片付けは省略する

4. **後片付け不要**: データ変更がないため後片付けは不要

## 現行環境と移行後環境の同一前提の保証

### 方針

現行環境と移行後環境で同一のFlywayシードデータ（V020〜V025）を使用することで、同一の前提データを保証する。

| 項目 | 現行環境 | 移行後環境 | 同一性保証方法 |
|---|---|---|---|
| DBシードデータ | Flyway V020〜V026（WildFly起動時） | Flyway V020〜V026（Quarkus起動時） | 同一SQLファイルを使用 |
| Keycloakユーザー | proquip-realm.json（KC22） | proquip-realm.json（KC26） | レルム設定ファイルの内容を同期（ポート・URL以外） |
| テストユーザー | tests/e2e/fixtures/users.ts | 同一ファイルを使用 | ファイル共有 |
| テストデータバリエーション | spec/test-data.md（本ファイル） | 同一仕様に基づく | 本ファイルが信頼の原典 |

### E2Eテストのアダプターパターン

F-01で確立したアダプターファクトリーパターン（TARGET_ENV環境変数で現行/移行後を切替）を継続使用する。テストデータの期待値は共通で、セレクターのみがアダプターで差し替えられる。
